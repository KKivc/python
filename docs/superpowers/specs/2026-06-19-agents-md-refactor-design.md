# AGENTS.md 重构设计（方案 A：拆分 + 强化）

日期：2026-06-19
状态：已批准设计，待实施

## 背景

`AGENTS.md` 是编程 Agent **每次会话自动加载**的导师 prompt。当前文件（4134 字节，71 行）把两类生命周期完全不同的内容混在一起：

- **指令**（第 1-18 行）：稳定、要 Agent 严格执行
- **教学日志**（第 20-71 行）：不断增长的历史快照

混合导致三个问题，随项目数增长持续恶化。

## 评估结论（全维度体检）

| 维度 | 评分 | 核心问题 |
|---|---|---|
| 教学法 | 8.5/10 | 方法扎实，缺"间隔复习"和"卡住升级的时间维度" |
| 指令有效性 | 6/10 | 规则正确但约束太软；历史里的已解决坑被当成现行指令 |
| 文档结构 | 4/10 | 日志与指令混杂，信噪比随时间下降（核心病灶） |

## 目标

1. 把 Agent 每次加载的内容压缩为高信号的纯规则
2. 给最易被违反的规则（"不要直接给代码"）加强约束
3. 补两条教学法软规则（间隔复习、卡住超时升级）
4. 完整保留历史教学记录，供人回顾

## 方案

### 文件拆分

```
AGENTS.md      ← 只留规则（学习模式 + 教学规则 + 教学过程约束）
                 + 末尾一行指针指向 PROGRESS.md
PROGRESS.md    ← 搬入全部教学记录（进度、教学特点、各轮总结、坑点）
```

### AGENTS.md 改动细则

1. **保留**：学习模式、教学规则 1-5、教学过程约束 4 条——这些是稳定指令。
2. **强化关键规则**：给"不要直接输出代码"加 `IMPORTANT` 强调，并补明确触发边界：
   > 除非我明确说"给提示/给伪代码/给代码"，否则只回答我提出的具体问题，不要主动写实现代码。
3. **新增软规则两条**：
   - 间隔复习：每个新项目开始前，用一个旧概念的小问题热身。
   - 卡住超时升级：同一个点卡超过约 15 分钟，主动降一档提示（提示→伪代码→关键函数→完整代码）。
4. **末尾加指针**：`> 历史教学记录见 PROGRESS.md`

### PROGRESS.md 内容

原文件第 20-71 行原样搬入，标题层级调整为顶层（`# 教学进度与记录`）。内容不改写，只迁移。

## 不做什么（YAGNI）

- 不重写现有规则的措辞和风格（那是方案 C）
- 不重新组织 PROGRESS.md 的历史内容，只搬运
- 不动 test/ 目录的代码组织（题外）

## 验收标准

1. `AGENTS.md` 只含规则，篇幅明显缩短（目标 < 40 行）
2. 关键规则"不要直接给代码"带强约束 + 明确触发边界
3. 教学法两条软规则已加入
4. `PROGRESS.md` 完整包含原历史记录，无丢失
5. `AGENTS.md` 末尾有指向 `PROGRESS.md` 的指针

## 遗留提醒（不在本次范围）

- `AGENTS.md` 当前在 `.gitignore` 中被忽略，未纳入版本控制。作为规则文件这与其定位矛盾，建议后续决定是否入库。
